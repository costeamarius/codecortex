# ADR-003: Codecortex-enabled repositories define the agent operating model

- **Status:** Accepted
- **Date:** 2026-03-22

## Context

CodeCortex is designed as a repository memory system for AI agents, extended with a deterministic execution substrate.

Its value depends not only on providing repository-local capabilities, but also on ensuring that participating agents actually use those capabilities consistently.

A repository may be accessed from different environments:
- an IDE agent running inside the project workspace
- an OpenClaw agent operating from outside the IDE
- future automation agents or agent runners

If these environments behave differently when interacting with the same repository, the system becomes unreliable.

Examples of undesired divergence:
- one agent uses CodeCortex retrieval while another manually explores the repository
- one agent performs mutations through the execution layer while another edits files directly
- one agent respects repository-local rules while another treats CodeCortex as optional

This would break the goal of deterministic and consistent repository operation.

## Decision

If a repository is **Codecortex-enabled**, then that repository defines the operating model for all participating agents.

This means that any participating agent, regardless of runtime environment, must adopt the repository-defined CodeCortex workflow when interacting with that repository.

This includes agents running:
- inside an IDE
- through OpenClaw
- through future supported agent integrations

## Meaning of “Codecortex-enabled”

A repository is considered Codecortex-enabled when it contains the required local markers and runtime structure indicating that CodeCortex is installed and should be used.

Examples of enabling markers may include:
- a `codecortex/` directory
- a `.codecortex/` directory
- a valid repo-local CodeCortex CLI
- repository instructions generated by CodeCortex
- future explicit project config markers

The exact detection mechanism may evolve, but the architectural rule remains the same:

> when CodeCortex is present and configured for a repository, participating agents must use it.

## Operating model requirements

For a Codecortex-enabled repository, participating agents must:

1. use CodeCortex as the primary repository interface
2. prefer CodeCortex retrieval over redundant broad manual exploration
3. use the repo-local execution path for supported repository mutations
4. respect repository-defined execution constraints
5. align behavior across IDE and OpenClaw environments

## OpenClaw implication

OpenClaw is not only expected to be capable of running CodeCortex.
It is expected to become **Codecortex-aware**.

If OpenClaw detects that a repository is Codecortex-enabled, it must switch to the repository-defined operating model.

This means OpenClaw should:
- detect CodeCortex presence
- activate the appropriate workflow
- invoke the repo-local CLI
- avoid bypassing CodeCortex behavior with direct project logic

OpenClaw remains an integration and runner layer, not the owner of repository behavior.

## IDE implication

An IDE agent operating inside the repository must follow the same model.

The IDE environment does not define different behavior.
It only provides another execution environment for the same repository-defined rules.

## Principle

Repository behavior must be **repo-defined**, not **environment-defined**.

The same repository should behave consistently whether accessed from:
- an IDE
- OpenClaw
- another supported agent runtime

## Enforcement direction

This ADR does not require full enforcement machinery in v1, but it defines the direction clearly.

The system should evolve toward:
- repository detection
- activation of CodeCortex mode
- repo-local CLI usage
- reduced reliance on agent memory or informal convention
- eventual stronger enforcement for participating agents

## Consequences

### Positive

- consistent behavior across agent environments
- stronger alignment between IDE and OpenClaw workflows
- lower risk of repository interaction drift
- better basis for deterministic execution and coordination
- cleaner project ownership of behavior and rules

### Negative

- participating integrations must implement detection behavior
- some environments may require extra setup to respect repo-local workflow
- enforcement may initially be partial rather than absolute

## Non-goal

This ADR does not require CodeCortex to become mandatory for all repositories.

It only states:
- if a repository is Codecortex-enabled,
- participating agents must treat CodeCortex as the operating model for that repository.

## Final statement

CodeCortex does not merely provide optional repository tools.  
For Codecortex-enabled repositories, it defines the operating model that participating agents must follow.

# Phase 1 Boundaries

- **Status:** Accepted for implementation
- **Date:** 2026-03-22
- **Scope:** Boundary decisions for the first implementation cycle aligned with ADR-001, ADR-002, and ADR-003.

## Purpose

This document defines what is preserved, what is introduced, and what is intentionally deferred in the current CodeCortex implementation cycle.

It exists to keep v1 focused and to prevent architecture drift during execution-layer introduction.

## 2. Minimal execution v1 boundary

v1 introduces a **minimal deterministic execution layer**.

### Included in v1

- repo-local execution package under `codecortex/execution/`
- safe file mutation flow
- validation-before-commit
- structured operation logging
- minimal per-file write locking with TTL
- repo-local execution CLI commands
- machine-readable success / failure / blocked responses

### Explicit non-goals in v1

The following are deferred and are not part of the implementation target for this cycle:

- heartbeat-based lock renewal
- read/write lock separation
- multi-resource transactions
- deadlock handling
- automatic rollback orchestration
- advanced stale-lock recovery workflows
- conflict detection across complex concurrent workflows
- global execution layer
- OpenClaw-hosted execution logic

### Coordination position for v1

Advanced coordination is deferred.

v1 only introduces the minimum coordination necessary for correctness:
- one write lock per file
- lock TTL
- blocked responses when lock acquisition fails
- expired lock replacement during acquisition

## 3. Integration boundary

### OpenClaw role

OpenClaw remains:
- a runner
- a detector
- a bootstrapper
- an integration layer

OpenClaw must not become:
- the owner of execution logic
- the owner of file mutation logic
- the owner of locking logic
- the owner of repository behavior

### Execution path rule

The repo-local CLI is the only supported execution path for participating agents.

Target pattern:

```bash
cd /repo && python -m codecortex.cli <action>
```

or equivalent installed command if provided.

### Direct mutation rule

Participating agents must not mutate files directly when a supported CodeCortex execution path exists.

---

## 4. Standalone usability boundary

CodeCortex must remain usable without OpenClaw.

This means:
- repository memory features must work independently
- execution-layer v1 must work independently
- CLI usage must not require OpenClaw runtime
- OpenClaw is an integration target, not a runtime dependency

## 5. Agent operating model boundary (ADR-003)

### Codecortex-enabled repository definition for v1

For v1, a repository is considered Codecortex-enabled if one or more of the following are present:

- a repo-local `codecortex/` implementation directory
- a `.codecortex/` runtime directory
- a working repo-local CodeCortex CLI entrypoint
- repository instructions generated by CodeCortex

This detection model may later become stricter through explicit config markers.

### Minimal detection markers for v1

The minimal practical detection markers for this phase are:
- `codecortex/`
- `.codecortex/`
- CodeCortex-generated repository instructions

### Behavior when Codecortex mode is active

When a participating agent detects a Codecortex-enabled repository, it should:
- prefer CodeCortex retrieval over broad manual exploration
- use repo-local CodeCortex interfaces for supported operations
- avoid bypassing the execution layer for supported mutations
- align with repository-defined behavior rather than environment-specific behavior

### Architectural principle

Repository behavior is **repo-defined**, not **environment-defined**.

The same repository should expose the same operating model to:
- IDE agents
- OpenClaw agents
- future supported participating agents

---

## 6. Summary

v1 is intentionally narrow.

It preserves the current repository-memory product, adds a minimal deterministic execution layer, keeps OpenClaw as integration-only, and establishes the first practical version of a repository-defined operating model for participating agents.
